## How to run

Easiest way: play the build on itch.io - https://yargf.itch.io/space-must-burn

From source:

1. Clone the repo.
2. Open the project with Unity `6000.3.6f1` (Unity Hub ‚Üí Add ‚Üí select the cloned folder). First import takes a few minutes: Unity pulls packages and compiles Burst.
3. In the Project window navigate to `Assets/MyAssets/Scenes` and open `AiTest.unity`.
4. Press Play, use the demo buttons to spawn army presets and battles.

No extra setup is needed.

## Performance

The whole project is built around one goal: thousands of ships fighting at once without frame spikes.

- Stress test: 5 000 ships in one scene at 60+ FPS. Measured in a standalone build on Ryzen 7 7800X3D + RTX 4070 Ti; the test counts as passed only if FPS stays above 60.
- How it holds up: flow fields instead of per-unit A*, a spatial hash for every neighbor query, per-frame budgets with jittered retries for all heavy work, pooled projectiles instead of Instantiate/Destroy, and components written only on real changes so ECS change filtering keeps working. Details in [Design notes](#design-notes).

## How it works

Input turns clicks into commands. Commands sit in `DynamicBuffer` queues on ships and squads. ECS systems do all the game logic. UI reads state and sends commands back. Input and UI never touch game logic directly.

### Command pipeline

Right click goes like this: `CommandInputHandler` raycasts what you clicked (enemy = AttackTarget, friendly = FollowEntity, empty = MoveToPoint), collects selected squads from ships with `Selected`, and writes the command. If a whole strike group is selected the command goes as a group order, otherwise per squad. Ships without a squad get it in their own queue. Shift adds the command to the queue, without Shift the queue is cleared and the command runs now.

Commands sit in the buffer until the ship is ready. `CommandDequeueSystem` checks every frame: ship is Idle, guarding or arrived, then it takes the next command, sets ship state and asks for a flow field group. Squad commands go through `SquadCommandApplyUtility` which turns them into the same per-ship state.

### Flow field

A* would need a path per unit, too expensive with hundreds of ships. Flow field is one direction map per target, any number of ships can use it. A ship just checks the cell it stands on and gets the direction. One group of ships with a common target = one field.

The field is built in two passes. First a BFS wave from the target: target cell costs 0, its neighbors 1, their neighbors 2 and so on, walls are skipped, expensive cells cost more. Second pass gives every cell a direction to its cheapest neighbor. Diagonal directions are allowed only when both side cells are free, so ships don't cut corners through walls.

Details:

- Three walkability layers (small/medium/large). A big ship doesn't fit where a fighter does, so on the Large layer narrow passages count as walls. A group builds its field on its own layer.
- Budget: max 2 BFS rebuilds per frame. If you order 30 groups to move, the fields get built over several frames.
- Cache: if target cell, layer and obstacle version didn't change, the field is not rebuilt at all.
- Target inside a wall: `TryFindNearestWalkableCell` searches around in rings.

### Ship velocity

Final velocity is a sum of separate parts, each written by its own system: `PathVelocity` from the flow field, `CombatVelocity` from FightSystem, `AvoidanceVelocity` from obstacles, plus boid separation/alignment/cohesion from neighbors. `SetTotalVelocitySystem` sums it all and clamps by maxSpeed and acceleration (with EMP and buff multipliers), `MoveVelocitySystem` applies it to position.

Boids run as a parallel job (`IJobEntity`). Small ships update every second frame, big ships every frame but separation only. Avoidance: the hard check (ship already inside a bad cell) runs every frame, the soft preventive sampling runs in stagger buckets.

### Spatial hash

`ShipToGridSystem` rebuilds `NativeParallelMultiHashMap<int2, Grid>` every frame: the world is cut into cells (6 units, plus a 30 unit grid for big ships), each cell holds the ships inside it. "Who is near me" then means checking a few neighbor cells instead of every ship on the map. Every neighbor search in the game goes through these maps: boids, targeting, bullet hits, aggro.

### Targeting

There are two levels of targets. `ShipAgroSystem` picks the target of the ship (who I fight, where I fly in combat). `EmbeddedFindTargetSystem` picks a target for every turret slot separately. So a ship can fly at a battleship while its turrets shoot fighters that are closer.

Turret search is written so it doesn't spike the frame:

- `findTargetTimer` is an absolute timestamp. Timer not due = cheap path, only check the current target is alive. Due = heavy path with a grid scan.
- Hard budget: max grid scans per frame. Out of budget = keep the old target, retry a bit later. Retry delays are jittered per slot so thousands of turrets don't wake up on the same frame.
- Pressure: a map of target to how much dps is already assigned to it. A candidate that half the fleet already shoots at gets a penalty. Without this every turret picks the same closest target and most damage is overkill.

### Fight patterns

`FightSystem` runs for ships in combat and writes one thing: the point where the ship should fly (`fightTarget`). Speed and turning is movement's job. Each ship has a pattern from its prefab: HoldDistance, CloseAndHold, Orbit, AttackRun, InterceptorPass, MissileAttackRun, Strafe, Dogfight, Swarm.

Patterns with phases work like a small state machine. AttackRun for example: approach, firing pass through the target, breakaway, reposition, and again. Every ship seeds its own rng from the entity id, so twenty fighters don't do identical maneuvers in sync. The ideal distance is always clamped to real weapon range, otherwise a pattern can keep the ship where it never fires.

### Weapons and bullets

Weapon stats live in ScriptableObjects. At load `WeaponProfileDatabaseBuildSystem` bakes them into a BlobAsset database: one immutable copy in memory, readable from Burst jobs, referenced by index. The fire flow is three steps: `EmbeddedWeaponFireRequestBuildSystem` puts a request into a queue when a slot is ready, `BallisticFireRequestExecutionSystem` takes a bullet from the pool and launches it, `BulletMoverSystem` moves it and checks hits.

Fast bullets can jump over a ship between two frames, so the hit check is a segment from the old position to the new one against ship AABBs, candidates come from the spatial hash cells along the way. Closest hit on the segment wins.

### Projectile pool

Bullets are never created or destroyed during battle. The pool pre-spawns them, disabled and parked outside the map. Acquire pops from a free list, validates the entity is still sane, and can grow the pool up to a hard cap. Pool full = the shot is dropped and counted. Release has double-return protection, otherwise one bullet could end up in the free list twice and get handed to two shooters. The pool counts drops, double releases and peak usage, after a battle these counters tell you if the pool size is right.

### Squads and strike groups

Three levels: strike group (player hotkey group, Ctrl+1..9), squad (flies in formation, has a command queue), ship (slot 0 is the leader). Group orders resolve into squad commands, squad commands into per-ship state.

Formations: every ship knows its slot among alive members, `FormationUtility` gives the offset from the leader (wedge, line, ring, column), the offset rotates with squad heading. The whole squad flies on one flow field to the squad anchor, formation is just the final correction. Offsets are cached and rebuilt only when formation, spacing or member count changes.

Carrier squadrons are the same squads with `origin = Carrier`. The player can't command them directly, they are controlled by carrier logic and hangar slots.

### UI

UI gets state through snapshots. `UiSelectionQueryUtility` walks the selection once and builds one struct (counts, fire/move modes, mixed flags), every panel renders from it. Buttons call `CommandInputHandler` methods, so a UI click goes through the same pipeline as a click on the map.
